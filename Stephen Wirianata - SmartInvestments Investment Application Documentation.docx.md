# 4\. Producing and Implementing {#4.-producing-and-implementing}

## 4.1 Development Process {#4.1-development-process}

SmartInvestments address the solution by constructing a simulated investment environment designed to provide the fundamentals of investing in a safe setting. SmartInvestments was developed in Visual Studio Code using Python, supported by several libraries and an external API handling the graphical interface, functionality, and data retrieval. Development followed the agile methodology, essentially based on the project's iterative nature and relying heavily on user interaction, frequent updates, and feedback. The construction and testing features in short cycles presented better usability and logic issues to be identified earlier rather than later in the development process. CustomTkinter was the main library employed in the creation of the graphical interface, providing a modern look with rounded buttons and styled entry fields without building custom styles from scratch. Pandas managed financial data from the API, simplifying filtering, sorting for calculating portfolio balances and percentage changes. Matplotlib visualised this data through line graphs, helping beginner investors visually data easily. The Alpha Vantage API supplied real stock market data for authenticity, while some UI styling code was adapted from GitHub and YouTube to save development time. The Alpha Vantage API was later changed to TwelveData API because the previous api had issues with daily restrictions and rate limiting while the newer api presented better statistics and efficiency capabilities.

SmartInvestments was constructed around a class-based, modular design, where every portfolio has its own class, reflecting the object-oriented principle of encapsulation and made the codebase easier to manage, allowing the rest of the program to interact with a portfolio object without needing to know its internal details with only the main portfolio located without the class structure. This modular separation improved maintainability, increased flexibility, productivity and improved testing, effectively identifying changes to portfolio logic without affecting multiple portfolios and tracing bugs and errors more efficiently. The modular design presented further extension capabilities for the future, supporting new multiple portfolio opportunities and minimising the redesign of the whole structure. Modelling portfolios as classes reflected real-world concepts directly, making the program more intuitive to read and debug while keeping each portfolio instance independent. SmartInvestments depended on the TwelveData for live data, posing vulnerability issues, like network failures, rate limits, or invalid ticker symbols. To manage potential issues, try and accept functions were applied in all API calls, ensuring failed or unexpected responses were handled gracefully instead of the program crashing. This was important for an educational tool aimed at non-technical users ensuring a clear, handled error provides a usable simulation when external data is temporarily unavailable. 

---

## 4.2 Key Features Developed {#4.2-key-features-developed}

When building SmartInvestments, I established the importance of creating multiple investment functionalities, ensuring the investment application was engaging and interesting to reflect real-world investing. The core features allow users to interact with several different financial investment opportunities, compete with other users through a leaderboard, and view their portfolio performance through graphs and labelled statistics. 

The high yield savings account functionality allows users to deposit a sum of money into a savings portfolio, enabling passively compounding at a 5% annual interest rate until the money is withdrawn. Users are enabled to deposit or withdraw any amount at any time.The Certificate of Deposit (COD) similarly permits users to deposit money into a separate portfolio, where it earns interest over a fixed time period. Unlike the savings account, users can name their CD investment and calculate returns, but are restricted from withdrawing funds until the selected time period ends. The index fund and stock market functionalities worked similarly. The index fund functionality made users experience real-world trading through a fund tracking a broader market index, highlighting its secure safety for the long-term over single stocks while the stock market functionality presents individual stocks, letting users experience the higher risk and volatility of trading individual equities. Together, these features were included to expose users to a fuller spectrum of investment risk, from the stability of fixed-term deposits to the unpredictability of individual stocks. The leaderboard was strategically implemented to introduce friendly competition between users, encouraging continued engagement, while the graphs and statistics provided users with a clear, visual understanding of how their decisions affect their portfolio over time. 

---

## 4.2.1 Back-End Engineering Contribution {#4.2.1-back-end-engineering-contribution}

The back-end engineering methodology of SmartInvestments performed an essential part in ensuring the application's reliability and usability and enabling data processing  validation, storage, and authentication. For data processing, the TwelveData API was utilised to retrieve real stock market data, processed using Pandas to calculate portfolio balances and percentage changes, allowing the front-end to effectively display accurate, current figures without users needing to perform calculations themselves. Validation and logic were primarily established for the TwelveData api through many try and except blocks ensuring that failed requests or invalid responses were effectively caught and managed gracefully rather than crashing the application. This back-end logic was essential to maintaining a smooth user experience and increasing the efficiency of the program. Storage and retrieval evolved significantly over development. In the fourth sprint, user and portfolio data was stored using the os module, managing data through plain files and folders, before it was replaced in the fifth sprint with JSON, providing a more structured and reliable way to store and retrieve nested data, such as multiple portfolios and their transaction histories, making the storage system considerably easier to maintain and extend. Authentication and authorisation was implemented through a security login system, allowing each user's data, including their portfolios and progress, to be saved and retrieved individually. The importance of the security login system established a personalised experience, where users could return to the application and continue instead of restarting every time they started a new session.

## 6.2 Secure Software Design and Data Handling {#6.2-secure-software-design-and-data-handling}

SmartInvestments was reasonably effective when it came to data utilisation. In terms of secure coding practices, the most notable measure applied was the consistent employment of try and except functions in API requests, ensuring failed requests, invalid ticker symbols, or network issues didn’t crash the application or expose sensitive information. This approach reflected an importance in the secure coding practices, ensuring the program fails gracefully rather than destructively and would operate efficiently. However, some input validation entries required increased thorough validation, ensuring entry fields were explicitly checked before being processed, which represents a major issue in the application's defensive design. Without a strong validation system, the investment application is highly exposed to unexpected behaviour. I believed my program also succeeded in hashing the password of users, through the deployment of a hashlib module, establishing safe data handling while ensuring the data safety and privacy of users. Although password hashing was implemented, no reliable encryption was used for the JSON data files. Future versions should encrypt sensitive user data to provide an additional layer of protection.  Data storage was extensively dependent on JSON files to store user credentials and portfolio information, evolved from an earlier os-based file system utilised in the fourth sprint. The importance of the data storage transition reduced load time and file errors and improved the safety and privacy of user data. The program had implemented a security authentication login system, ensuring only authorised users would access the investment application, minimising the risk of data exposure and improving user engagement. Most of the program had been developed to handle errors through try and except functions and if statements, enabling reliable error handling if potential issues occurred. The API error handling supports reliability and improves user trust by preventing crashes during normal iterations, however the lack of input validation highlights clear areas for improvement in future development. The API key was securely stored in a dotenv file to protect developer credentials and prevent data breaches.
